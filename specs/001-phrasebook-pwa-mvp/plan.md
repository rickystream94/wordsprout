# Implementation Plan: WordSprout — Personal Vocabulary Phrasebook (MVP)

**Branch**: `001-phrasebook-pwa-mvp` | **Date**: 2026-04-12 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/001-phrasebook-pwa-mvp/spec.md`

## Summary

Build an offline-first Progressive Web App that allows language learners to capture,
organise, search, and review vocabulary encountered in real life. Users maintain multiple
phrasebooks (one per language pair, with ISO 639-1 source and target languages). Core
entry creation, search, and browsing work fully offline via IndexedDB + MiniSearch. A
replay-queue pattern synchronises local changes to Azure Cosmos DB when connectivity
resumes. On-demand AI enrichment (GPT-4o-mini via Azure AI Foundry) is rate-limited and
always user-triggered. Access is invite-only; allow-list enforcement is applied
server-side on every authenticated Azure Functions request via JWT validation + Cosmos
DB point-read. All user input and AI output is sanitised before storage and display.

## Technical Context

**Language/Version**: TypeScript 5.x (frontend + API)
**Primary Dependencies**: React 18, Vite, Dexie.js, MiniSearch, MSAL.js v3; Azure Functions Node.js v4, @azure/cosmos, jsonwebtoken, jwks-rsa, isomorphic-dompurify
**Storage**: IndexedDB via Dexie.js (client-side); Azure Cosmos DB Serverless Core/NoSQL API (server-side)
**Testing**: Vitest + React Testing Library (frontend); Vitest (API unit); integration tests against Cosmos DB emulator
**Target Platform**: Modern evergreen browsers (desktop + mobile); installable PWA (iOS Safari, Android Chrome)
**Project Type**: web-app (React PWA frontend + Azure Functions serverless API)
**Performance Goals**: Full-text search <1s for 2,000 entries offline; AI enrichment response <10s online
**Constraints**: Offline-capable; OWASP Top 10 compliance; $100/month absolute ceiling; all AI calls user-triggered; three environment stages: local (fully mocked, no Azure), dev (local app + real Azure dev resources), prod (fully cloud-hosted)
**Scale/Scope**: Up to 50 private-preview users; ~2,000 entries/user; ~10 writes + 50 reads per user/day

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-checked after Phase 1 design — all ✅.*

- [x] **I. Encounter-First** ✅ — No app-imposed vocabulary. All entries are user-initiated. No word-of-the-day, auto-lists, or mandated content anywhere in the design.
- [x] **II. Learner Ownership** ✅ — Every field on every entry is user-editable at any time. AI enrichment fields are editable proposals; user edits overwrite AI values with no preserved original.
- [x] **III. Offline-First** ✅ (NON-NEGOTIABLE) — IndexedDB is the primary data store; all P1 (entry capture), P2 (search/browse), P4 (tags/PoS), P6 (review/state) stories work without network. Sync is background-only via online-event replay queue.
- [x] **IV. AI as Assistant** ✅ — AI enrichment is explicitly user-triggered per entry (FR-021). Daily per-user quota enforced server-side (FR-024). No automatic or background AI calls. AI not involved in auth or security paths.
- [x] **V. Cost-Conscious** ✅ — Serverless-first stack. Estimated ~$1–5/month for private preview (50 users). Cosmos DB Serverless ~$0.05/month; Functions within free tier; AI Foundry GPT-4o-mini ~$1–5/month with rate limiting. Well under $20 target and $100 ceiling.

tooling in MVP — each root has its own `package.json`. Shared TypeScript types are
co-located in `api/src/models/` and referenced from the frontend via path alias or copy
until a shared package is justified.
